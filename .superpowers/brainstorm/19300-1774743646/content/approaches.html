Dashboard Architecture Approaches

How the dashboard connects to your existing bot

A

Embedded API Server

Add an Express/Fastify HTTP API inside the existing bot process. React frontend served as static files from the same process.

Pros

  • Direct access to in-memory state (tracked orders, prices, ads)
  • Single process to manage — no IPC or DB polling
  • Real-time updates via WebSocket from the EventBus
  • Simplest deployment

Cons

  • Crash in HTTP layer could take down the bot
  • Adds weight to the main process
  • Harder to restart dashboard without restarting bot
B

Separate Process, Shared SQLite

Dashboard runs as its own Node process. Reads from the same SQLite DB (WAL mode supports concurrent readers). Bot exposes a tiny HTTP API only for actions (release, cancel).

Pros

  • Dashboard crashes don't affect the bot
  • Can restart/redeploy independently
  • SQLite WAL mode already supports concurrent reads
  • Clean separation of concerns

Cons

  • No real-time in-memory state — must poll DB or add a small WS bridge
  • Two processes to manage
  • Action endpoints still need to reach the bot process
C

Hybrid: Embedded API + Separate Frontend Dev Server

Bot gets a lightweight API layer (Express + WebSocket). React app developed separately with Vite, proxies API calls to bot. In production, built React files served by the bot.

Pros

  • Best DX — Vite HMR for frontend, bot runs independently
  • Real-time via WebSocket from EventBus
  • Single deployment in production (static files served by bot)
  • Clean dev workflow: frontend and bot concerns separated

Cons

  • Still couples API to bot process (same crash risk as A)
  • Slightly more setup (Vite proxy config)
